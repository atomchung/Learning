# 找出你的「未知」：跟 Claude/Fable 協作時的引導框架

來源：Thariq（Anthropic）在 X 上發的實戰心得，用戶提供繁中翻譯全文（2026-07-05 讀）。屬於**個人心法/工作流分享**，不是實測數據——照攝取節奏的兩篩：過「信號 vs 表面數字」（是），但屬於「經驗談」而非「量測」，當作可試的技巧集，不當鐵律。

## 一句話

模型能力越強，瓶頸越往回移到「使用者能不能把自己的未知講清楚」；這篇提供一套詞彙（四種未知）+ 六個專案階段的 HTML artifact 技巧，用來把「講不清楚」的東西提前挖出來。

## 核心框架：四種未知

借用 Rumsfeld 的「已知/未知」矩陣，套進跟 Claude 協作的情境：

- **已知圈的已知**：寫進 prompt 裡、講得出來的部分。
- **已知圈的未知**：自己知道還沒想清楚的部分。
- **未知圈的已知**：自己心裡有判準、但講不出來，一看到就能認出對錯（品味型知識）。
- **未知圈的未知**：連「這是個問題」都沒意識到的部分。

厲害的人不是天生 prompt 寫得好，是**未知本來就比較少**——跟程式庫、跟模型行為模式同步度高。減少/規劃未知本身是一項可練的技能。

## 六個階段的技巧（都是拿 HTML artifact 當引導載體）

**執行前**
1. **盲點檢視**：明講自己在這塊的背景很淺，請 Claude 主動列出「這題的未知圈的未知」，反過來教你該問什麼問題。
2. **腦力激盪/原型**：對「未知圈的已知」（講不出來但看了就知道對不對的東西，通常是視覺/UX）先出多個方向的假資料原型，比早期就綁死實作路徑便宜很多——規格微調在原型階段代價低，進了 code 就貴。
3. **訪談**：請 Claude 針對還模糊的地方**一次問一題**反過來問你，並指定「優先問答案不同會改變架構的問題」。
4. **參考資料**：講不清楚就給範例——最好的參考永遠是原始碼/既有實作，即使語言不同也能讀出行為語意。
5. **實作計畫**：請 Claude 產出計畫時，把「最可能被你調整的決策」（資料模型、型別介面、UX 流程）放最前面審查，機械式重構放最後（這段直接信任）。

**執行中**
6. **實作筆記**：請 Claude 維護一份暫時的 decision log，記錄它被迫偏離原計畫的邊界情況（遇到就選保守選項、記下來、繼續做），下一輪能從中學。

**執行後**
7. **提案/說明文件**：把原型+規格+實作筆記打包成一份文件給團隊看，讓審查者從跟你一樣的未知起點看起，加速理解跟核准。
8. **隨堂測驗**：請 Claude 針對這次變更出一份測驗，自訂規矩「答對才 merge」——單看 diff 只有表面理解，很多行為取決於既有 code path。

## 對我的判斷

**呼應既有卡片**：這篇的核心訴求（瓶頸從模型移回人）跟已沉澱的「個人攝取流程——資訊不稀缺，判斷與校準才稀缺」（`notes/info-intake-routine.md`）同構，只是換了個場景（讀資訊 vs 寫 code/設計）。也呼應 `topics/coding-agents/cards/eval-bottleneck-is-criteria-not-tooling.md`：講不清楚判準，本質也是一種「未知圈的已知」講不出來。

**可以直接借用到這個 repo 自己的工作流**：
- 開新主題時「先問清楚邊界」（CLAUDE.md 既有規則）其實就是第 3 招「訪談」的簡化版；可以升級成「一次問一題、優先問答案不同會改架構的問題」。
- `/record` 產的任務記錄，性質上接近第 7 招「提案/說明文件」——已經在做,只是沒特別點名。
- 第 8 招「隨堂測驗」是我目前**沒做**的一塊：卡片化/長文產出後，沒有機制驗證使用者真的吸收了核心判斷,而不是我自以為講清楚了。可以在升級卡片或寫長文後，主動問幾題來對答案。

**保留/反例**：這是單一作者的個人心法文，沒有量化驗證「用了這套技巧輸出品質提升多少」，六個階段的分法也可能因人而異（例如「訪談」跟「盲點檢視」實務上常常混在一起做）。當工具箱心法用，不當流程規範套。

**要不要升卡？** 暫不升級——目前只出現這一次，且核心判斷（瓶頸移回人）已經有既有卡覆蓋，這篇的增量價值主要是「六階段 + HTML artifact 引導技巧」這個具體操作清單，屬於筆記層就夠（若「隨堂測驗」這招之後在別的脈絡重複出現 ≥3 次，再考慮拆成獨立卡）。
