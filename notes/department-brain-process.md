# 部門大腦：一堆 wiki 怎麼搭成會複利的知識庫

> 延續 [rag-vs-llm-wiki](./rag-vs-llm-wiki.md) 的團隊路線，把原則落成可執行流程。
> 前提（用戶確認）：**人＋agent 都要讀**、既有 wiki 散在 **Confluence/Notion 量大很多過期**、最痛點是 **沒人寫/沒人維護**。
> **關鍵澄清（2026-06-17）**：現況是**根本還沒有一個完整知識庫**。所以這不是「整理成熟 wiki」，是「從散亂中長出第一版」。三步走：**① 整理地圖 → ② 梳理 high-level learning → ③ 持續維護**。下面流程依此排序。

## 三個交付物（先想清楚要產出什麼）

不是「一個搜尋系統」，是三層由上而下的東西：

1. **地圖**：一張跨主題目錄，盤點部門「有哪些知識領域、每塊現況（有/缺/過期）」。對應這個 repo 的 `README.md`。**先有地圖才知道缺什麼。**
2. **High-level learning 層**：每個領域一份**高層綜述**——這塊我們的核心判斷/結論是什麼，不是堆碎片頁。對應這個 repo 的 `cards` 判斷層 + 長文。**這層才是「共享判斷」，是團隊真正缺的。**
3. **底層 entity pages + 來源**：細節頁與唯讀原料。對應 `notes/` + `raw/`。**按需長出，不求一次填滿。**

順序很重要：**地圖 → 綜述 → 細節**，由上而下。多數失敗的 wiki 是反過來——堆了一堆細節頁卻沒有地圖和綜述，所以沒人找得到、也看不出全貌。

## 先把問題診斷對（不然整套白做）

痛點是「沒人寫/沒人維護」=**動機與所有權問題**，不是檢索問題。所以三件事是陷阱，別做：

- ❌ 先上向量庫 / RAG —— 解的是「搜得到」，你缺的是「有人寫」
- ❌ 開會請大家「以後記得更新 wiki」—— 靠紀律的維護一定爛尾
- ❌ 做一個更漂亮的搜尋介面 —— 漂亮的空庫還是空庫

**勝負手**：把寫入的邊際成本壓到接近零 + 把維護外包給 AI。下面整套流程都繞這兩件事轉。

## 四條原則（從個人 repo 搬到部門）

1. **寫入是工作的副產品**，不是額外任務 —— 捕捉點設在工作已經發生的地方
2. **維護是 AI 的活**，不是人的紀律 —— lint / freshness / 矛盾偵測自動跑
3. **低門檻先進主庫** —— 扁平一段話就能進，別等「完整整理好」才寫
4. **單一正本所在地** —— 一個 git repo + 一份 schema；既有 Confluence/Notion 降為唯讀來源

## 流程：搭建 + 運轉

### Phase 0 — 立地基（一次性）

- **選正本所在地**：一個 git repo（純 markdown）+ 一份部門版 `CLAUDE.md`（schema）。
  - 為什麼是 repo 不是 Notion：agent 讀寫友善、版本控制=天然 audit、**PR = 寫入協作的單位**（解並發衝突）。
- **既有 Confluence/Notion 不搬不刪**：整批列為 `raw/` 來源層（唯讀考古層）。它們是原料，不是成品。

### Phase 1 — 從零長出第一版：先地圖、再綜述、後細節

現況沒有完整 KB，所以這段是重點。**由上而下三步**，別跳過前兩步直接堆頁：

**1-A 先畫地圖（半天～一天）**
- 讓 agent 掃既有 Confluence/Notion + 問幾個資深成員，產出一張**跨主題地圖**：列出領域、每塊標現況（已有/缺/過期/打架）。
- 這張地圖就是部門版 `README.md`，也是後續一切的導航列。人驗收這張就好。

**1-B 每塊梳理 high-level learning（一塊一兩小時）**
- 對地圖上每個領域，讓 agent 把散落的舊內容**綜述成一份「高層判斷頁」**：這塊我們的結論是什麼、有哪些已知坑、未解問題。
- 這層是團隊真正缺的「共享判斷」。**先把核心 5–8 個領域的綜述做出來**，比做 200 頁細節有用得多。

**1-C 細節頁按需長**
- entity pages 不一次填滿。等有人實際 query 到、或寫入流程帶進來，才編譯該頁（含 `freshness` + 來源連回 raw）。
- 別叫任何人去手工搬舊 wiki（那就是當初沒人做的事），讓 agent 按需編譯。

### Phase 2 — 寫入流程：讓「寫」變成副產品

捕捉點設在工作發生處，目標是**人不必開新習慣**：

- PR review 的結論、會議決議、Slack 上的解答、事故覆盤 → 丟進一個低門檻 `inbox`（append 一段話即可，不要求格式）
- 理想是接 bot：Slack 一個 emoji / PR 一個標籤就把該段落收進 inbox
- 每週 AI 把 inbox 綜合進 wiki（對應個人 repo 的 weekly-synthesis）

**所有權設計（解你最痛點的關鍵）**：每個主題有一個 owner，但 owner 的職責是**審核 AI 的編譯，不是自己寫**。人審不寫 → 維護負擔掉一個量級 → 才有人願意當 owner。

### Phase 3 — 維護流程：全部外包給 AI

- **freshness + lint**：過期頁自動標記、降權、提醒 owner，不靠人記得清
- **矛盾偵測**：AI 定期掃重複/打架的頁，整理成清單請 owner 裁決（解「各寫各的、打架」的前身）
- **meta-review**：定期把「哪裡該撈沒撈到、給錯答」轉成 schema 規則 —— 系統自己變聰明

### Phase 4 — 何時才上 RAG（多半不用急）

只有當「要搜大量**原文**」（會議逐字稿、PDF、Slack 歷史）塞不下 context 時才加，而且：

- 用**受管** RAG（Glean / Notion AI / Gemini File Search），別自建向量庫
- 只接在 `raw/` 層之上；**wiki 層永遠是預編譯的判斷層**
- 一句話：RAG 讓團隊「搜得到原文」，wiki 讓團隊「共享判斷」。你缺的是後者。

## 治理三難點（個人 wiki 沒有的）

- **並發寫入** → 用 PR 當寫入單位，AI 改、owner merge
- **新鮮度** → freshness frontmatter + lint，過期自動降權
- **權限** → 小範圍先全開放，別過早分層做權限系統（過度工程）

## 落地建議：別一次上整個部門

先挑**一個小組、一個高頻痛點主題**做 4 週 pilot：
1. 第 1 週：立 repo + schema → **畫出地圖** + **5–8 個領域的 high-level 綜述**，人驗收
2. 第 2–3 週：接一個捕捉點（先 PR 或會議擇一），跑 inbox→週綜合，細節頁按需長
3. 第 4 週：看 owner 審核負擔是否夠低、AI 綜述是否夠準 → 再決定擴散

驗收指標不是「庫有多大」，是兩件事：**①地圖+綜述讓人三分鐘看懂全貌 ②寫入零負擔、過期有 AI 在管**。這兩個成立，複利才會自己滾。
